System Architecture Document

Vibe Code → Production
via Autonomous Hardening Loops

A system that takes working-but-fragile prototypes and autonomously hardens them into production-grade software — fusing Karpathy's autoresearch pattern with Hamel Husain's eval-driven methodology to close the gap between "it works on my machine" and "it runs at scale with SLAs."

Active Architecture v1.0 — March 2026 Target: Vibe-coded prototypes → Production deploys
01 — Core Thesis

The Vibe Code Gap Is an Eval Problem

Architecture Insight

Vibe coding gets you to "it works" fast. But "it works" and "it's production-ready" are separated by hundreds of specific, discoverable defects — missing error handling, unvalidated inputs, no observability, implicit security assumptions, zero test coverage. Each defect is binary: present or absent. That makes this an eval problem. And eval problems have autonomous solutions.

The original Karpathy + Hamel system optimizes a single metric (model performance) via autonomous experimentation. Reframed for production hardening, the system optimizes across multiple readiness dimensions simultaneously — each with its own failure taxonomy, eval judges, and hard gates. The agent doesn't improve one number. It closes a checklist of hundreds of specific production defects, autonomously, one commit at a time.

Target file Single model/prompt file Any file in the prototype repo
Primary metric val_bpb (single scalar) Production Readiness Score (composite)

Hard gates No regression on val metric No broken tests + no new vulnerabilities + existing functionality preserved

02 — System Architecture

The Autonomous Hardening Loop

01

Defect Scan

Static analysis + LLM review of the entire prototype codebase. Produces a typed defect taxonomy: unhandled errors, missing validation, hardcoded secrets, no retry logic, implicit type coercions, missing indices, no rate limiting. Hamel's "theoretical saturation" — keep scanning until no new defect categories emerge.

02

Eval Suite

L1: Deterministic assertions — tests pass, types check, linter clean, no secrets in source, schema validation. L2: LLM judges — "Does this error handler cover the failure mode?", "Is this input validated against the schema?", binary pass/fail per defect. L3: Integration smoke tests on significant changes.

03

program.md

Karpathy-style instruction doc scoped to one hardening dimension at a time. Specifies: which files the agent may touch, what the defect is, what "fixed" looks like, which eval commands to run, and the time budget per fix attempt.

04

Agent Loop

Agent reads program.md, picks the highest-priority unfixed defect, applies a fix to the scoped files, runs the full eval harness. Single-defect-per-commit discipline — atomic, reviewable, revertable.

05

Gate

Pass ALL evals → git commit. Fail ANY → git checkout + log reason. Existing tests must not break (behavioral preservation). No exceptions — the agent cannot disable gates.

06

Feedback

Failed fixes feed back into the defect taxonomy. Human reviews committed batches at morning standup cadence. Readiness score updates on every commit. Loop terminates when all dimensions pass production threshold.

03 — Hardening Dimensions

What the Agent Actually Fixes

Each dimension gets its own program.md, its own eval judges, and its own pass threshold. The agent works one dimension at a time — depth before breadth.

Scan → Fix

Error Handling & Resilience

Unhandled promise rejections, missing try/catch, no retry logic on network calls, swallowed errors, missing timeout configurations, no circuit breakers. L1 gate: zero uncaught exceptions in test suite. L2 judge: "Does every external call have explicit error handling with meaningful error types?"

Hard Gate

Input Validation & Type Safety

Unvalidated API inputs, any types, implicit coercions, missing Zod/Pydantic schemas on boundaries, SQL injection vectors, path traversal. L1 gate: tsc --strict passes with zero errors. L2 judge: "Is every external input validated before use with a runtime schema?"

Fix

Observability & Debugging

No structured logging, missing request IDs, no trace correlation, console.log as the observability strategy, no health endpoints, no metrics. L1 gate: every API endpoint emits structured log on entry and exit. L2 judge: "Can a production incident on this endpoint be debugged from logs alone?"

Governance

Security Posture

Hardcoded secrets, missing auth checks, overly permissive CORS, no rate limiting, unencrypted data at rest, missing CSP headers. L1 gate: zero secrets in source (gitleaks), dependency audit clean. L2 judge: "Does this endpoint enforce authentication and authorization before any data access?"

Scan → Fix

Test Coverage & Behavioral Contracts

Zero tests, untested edge cases, no integration tests, missing fixture data, flaky tests. L1 gate: coverage above threshold (start at 40%, ratchet up per sprint). L2 judge: "Does the test for this function cover the documented happy path, error path, and edge cases?"

Hard Gate

Data Integrity & Persistence

Missing migrations, no foreign key constraints, unindexed query patterns, no backup strategy, race conditions on concurrent writes. L1 gate: all migrations reversible, schema matches ORM models. L2 judge: "Does this write path handle concurrent access correctly?"

04 — Eval Methodology

Hamel's Playbook, Applied to Code Quality

The core insight transfers directly: don't write evals for imagined defects — scan the actual codebase and write evals for defects you find. Vibe-coded projects have a finite (if large) defect surface. Scan it, categorize it, then build judges that catch each category.

Principle 1: Start with Defect Scan, Not Linting Rules

Manually review the prototype's critical paths. Categorize every production risk you find — not what you imagine might break, but what you can see is broken or missing. Open-code until you hit saturation (no new defect types in the last 20 files reviewed).

Principle 2: Binary Per Defect, Not Quality Scores

"Is this endpoint rate-limited?" is a binary. "How good is the error handling?" is not. Decompose every fuzzy quality concern into specific binary assertions. Track "5 of 8 API endpoints have input validation" as 8 separate binary checks.

Principle 3: Behavioral Preservation Is the Meta-Gate

The hardest constraint in production hardening: don't break what already works. Before the agent touches anything, capture the prototype's existing behavior as a test suite — even if it's just snapshot tests of API responses. This becomes the regression gate that every subsequent fix must pass.

Principle 4: The Domain Expert Is You

For vibe-coded projects, the person who built it is the domain expert. They know which shortcuts they took, where the dragons are, what "works" means. Their manual defect triage in Phase 1 is the highest-leverage 2 hours in the entire process.

Principle 5: Ratchet, Don't Threshold

Start with whatever coverage/quality baseline the prototype has. Set the gate at "don't go below current." Every committed fix ratchets the baseline up. The system monotonically improves — it can never regress.

Security Constraint

Non-negotiable gates for autonomous hardening. Every fix attempt must verify: no secrets introduced in source, no new network calls without allowlist, existing auth checks not bypassed, no data exposure in error messages, dependency additions scanned for known vulnerabilities. These are L1 hard gates — they block commits regardless of how much the fix improves other dimensions. The agent cannot modify the eval harness. run_evals.sh is read-only.

05 — Concrete Implementation

program.md for a Vibe-Coded Project

program.md — error handling hardening pass
# Vibe-to-Prod: Error Handling Hardening

## Target Scope
src/api/*.ts           ← agent may modify these files
src/services/*.ts      ← and these

## Fixed Budget
60 seconds per fix attempt (apply fix + run full eval suite)

## Primary Metric
error_handling_coverage: count(endpoints_with_proper_error_handling)
                       / count(total_endpoints)
Must be > current baseline. Monotonic ratchet.

## Hard Gates (L1)
- All existing tests pass          # behavioral preservation
- tsc --strict: zero errors         # type safety
- No secrets in diff (gitleaks)     # security
- No new any types introduced       # no shortcuts
- No console.log in production code # observability hygiene

## Soft Gates (L2)
- LLM judge: "Does the error handler for this endpoint:"
  · Catch specific error types (not bare catch)?
  · Return appropriate HTTP status codes?
  · Log structured error context?
  · Not leak internal details to the caller?
  Binary pass/fail per endpoint. Required: ≥ 85% pass rate.

## Experiment Protocol
1. Read defect list (sorted by priority)
2. Pick highest-priority unfixed defect
3. Apply minimal fix (smallest diff that resolves the defect)
4. Run: npm run eval:hardening
5. If ALL gates pass AND coverage improves → git commit -m "fix: [defect-id]"
6. Else → git checkout -- src/ && log failure reason
06 — Reference Implementation

File Structure

vibe-to-prod/ ├── programs/ # one per hardening dimension │ ├── error-handling.md │ ├── input-validation.md │ ├── observability.md │ ├── security.md │ ├── test-coverage.md │ └── data-integrity.md ├── evals/ # READ-ONLY to agent │ ├── harness.ts # orchestrates all eval levels │ ├── l1_assertions.ts # deterministic: tests, types, lints, secrets │ ├── l2_judges.ts # LLM binary judges per defect type │ ├── l2_judge_prompts/ # one prompt per defect category │ ├── behavioral_snapshots.json # captured from prototype before hardening │ ├── defect_taxonomy.json # categorized defects from Phase 1 scan │ └── security_gates.ts # secrets, auth, dep audit ├── logs/ # append-only audit trail │ └── fixes.jsonl # {defect_id, diff, results, δ_coverage} ├── scripts/ │ ├── run_fix.sh # agent calls this per fix attempt │ ├── scan_defects.ts # initial + periodic defect scanning │ ├── capture_behavior.ts # snapshot existing prototype behavior │ └── readiness_score.ts # composite score across all dimensions ├── target/ # THE VIBE-CODED PROJECT (agent works here) │ └── ... └── CLAUDE.md # agent boundary rules
run_fix.sh — core loop
#!/bin/bash
set -euo pipefail

# 1. Snapshot current state
BASELINE=$(git rev-parse HEAD)
BASELINE_SCORE=$(npx tsx scripts/readiness_score.ts)

# 2. Agent applies fix to target files (already done before this runs)

# 3. Run ALL eval levels
EVAL=$(npx tsx evals/harness.ts)

# 4. Parse gate results
L1_PASS=$(echo "$EVAL" | jq '.l1.passed')
L2_PASS=$(echo "$EVAL" | jq '.l2.passed')
BEHAVIOR_PRESERVED=$(echo "$EVAL" | jq '.behavioral.preserved')
NEW_SCORE=$(echo "$EVAL" | jq '.readiness_score')

# 5. Composite gate: ALL must pass + score must not regress
if [ "$L1_PASS" = "true" ] && [ "$L2_PASS" = "true" ] && \
   [ "$BEHAVIOR_PRESERVED" = "true" ] && \
   $(echo "$NEW_SCORE >= $BASELINE_SCORE" | bc -l); then
  git add target/
  git commit -m "fix: readiness $BASELINE_SCORE → $NEW_SCORE"
  echo "{\"status\":\"committed\",\"delta\":\"$(echo "$NEW_SCORE - $BASELINE_SCORE" | bc)\"}"
else
  git checkout -- target/
  echo "$EVAL" | jq '{status:"reverted",eval:.}' >> logs/fixes.jsonl
fi
07 — Anti-Patterns

What Will Kill This System

Anti-Pattern Why It's Fatal Mitigation
Fix everything at once Massive diffs are unreviewable, untestable, and impossible to revert cleanly Single-defect-per-commit. Atomic fixes only.
No behavioral baseline Agent "fixes" code by changing what it does — users see regressions Capture behavior snapshots before hardening begins. Gate every commit against them.
Linter-driven hardening Agent chases lint rules instead of real production risks. Busywork, not hardening. Defect scan from real traces / manual review. Evals follow observed risks.
Agent modifies eval harness Agent learns to game its own tests — marks defects as "fixed" by weakening the check Eval harness is read-only. Separate repo. Hash verification on every run.
Skip human triage Agent wastes cycles on low-value fixes. Priority order is wrong. Human does Phase 1 defect scan and sets priority order. Agent executes.
Quality score instead of binary Agent optimizes a fuzzy number instead of closing specific defects Binary per defect. "Is rate limiting present on /api/create?" — yes or no.
No audit trail Can't debug regressions, can't prove what was hardened, can't demo to stakeholders Append-only log: defect ID, diff, all eval results, readiness score delta.
08 — Success Metrics

How You Know It's Working

Leading

Fixes/Night

Target: 40+ fixes completing full eval pipeline per 8-hour overnight run. Below 20 = loop is too slow or fixes are too complex for atomic commits.

Leading

Commit Rate

Target: 20-35% of fix attempts pass all gates. Below 10% = defects need better decomposition. Above 50% = gates are too loose.

Lagging

Readiness Score Δ

Monotonically improving composite readiness score across committed fixes. Plateau = switch to next hardening dimension.

Safety

Behavioral Preservation

Zero regression in behavioral snapshot tests across all committed fixes. Any behavioral change → automatic revert + human review.

09 — Implementation Phases

Rollout Plan

Phase 1 — Day 1-2 · Scan + Baseline

Defect Scan & Behavioral Capture

  • Capture behavioral snapshots of the prototype (API responses, UI states, critical paths)
  • Manual defect scan: review every file in the critical path
  • Open-code defects into taxonomy (target: fewer than 10 primary categories)
  • Write L1 assertions for each defect category (deterministic, fast)
  • Prioritize: security → data integrity → error handling → everything else
Phase 2 — Day 3-4 · Eval Harness

Build the Gate

  • Build eval harness: run_evals.ts returning structured JSON
  • Version-control eval harness separately from agent-mutable code
  • Write binary LLM judge prompts for top 5 defect categories
  • Validate judges against manually-reviewed examples (≥0.85 precision + recall)
  • Implement behavioral regression gate against Phase 1 snapshots
Phase 3 — Day 5-7 · First Hardening Run

Autonomous Hardening

  • Write program.md for highest-priority dimension
  • Configure agent with read-only eval harness, mutable target scope
  • First overnight run: target 40+ fix attempts
  • Morning review: inspect commit log, readiness trajectory, revert reasons
  • Tune program.md constraints based on what the agent tried
Phase 4 — Week 2+ · Scale Across Dimensions

Parallel Hardening & Governance

  • Parallel agents on different hardening dimensions (git worktrees per dimension)
  • Readiness dashboard: composite score across all dimensions
  • Judge agreement monitoring (alert on drift above 5%)
  • Ratchet thresholds up as baseline improves
  • Promotion pipeline: hardened dev → staging smoke test → production deploy

The hardest part isn't the autonomous loop — that's just engineering. The hardest part is the first defect scan. Sit with the code. Find what's actually broken. Everything else follows from knowing what "production-ready" means for this specific project.
